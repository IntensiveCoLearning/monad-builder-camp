## 一、 Dapp 架构与开发流程

### 1\. 核心架构（极简四要素）

-   **前端 (UI)**：React/Vue + 钱包 (MetaMask) + 交互库 (Viem/Wagmi)。
    
    -   **读数据 (Read)**：通过 RPC 节点执行只读调用（如 `eth_call`），**免 Gas 费**。
        
    -   **写数据 (Write)**：构建交易 -> 用户钱包签名 -> RPC 广播上链，**扣除 Gas 费**。
        
-   **智能合约 (Contracts)**：部署在 EVM（以太坊虚拟机）上的 Solidity 代码，管理状态并执行去中心化逻辑。
    
-   **数据检索器 (Indexer)**：智能合约通过释放 `Event`（事件）记录日志。检索器（如 `Ponder`、`Subgraph`）捕获事件并写入传统数据库（PostgreSQL），供前端高效查询。
    
-   **去中心化存储**：IPFS / Arweave，存放非结构化数据（如 NFT 图像、元数据）。
    

### 2\. 标准开发生命周期

{需求确定} -> {合约编写与单元测试} -> {检索器(Indexer)编写} -> {前端与钱包集成} -> {测试网部署与审计} -> {主网发布}

## 四、 Solidity 核心语法极简通关

### 1\. 基础数据类型与默认值

-   `uint256`：256位无符号整数（默认值 `0`）
    
-   `address`：以太坊地址（默认值 `0x00...00`）
    
-   `mapping(address => uint256)`：键值对映射（**不可遍历，只能单点查询**）
    
-   `string` / `bytes`：动态字节数组/字符串（需声明存储位置为 `memory` 或 `calldata`）
    

### 2\. 可见性修饰符 (Visibility)

| 修饰符 | 内部调用 | 外部/用户调用 | 继承合约调用 | 备注 |
| --- | --- | --- | --- | --- |
| public | ✅ | ✅ | ✅ | 自动生成 getter 只读函数 |
| external | ❌ | ✅ | ❌ | 外部首选，比 public 节省 Gas |
| internal | ✅ | ❌ | ✅ | 类似 C++ 中的 protected |
| private | ✅ | ❌ | ❌ | 仅当前合约内有效 |

### 3\. 状态可变性修饰符 (State Mutability)

-   `view`：**只读**。读取链上数据，但不修改状态。
    
-   `pure`：**纯算法**。既不读取也不修改链上状态（如纯数学运算）。
    
-   `payable`：**可收钱**。允许该函数接收 ETH（可用 `msg.value` 读取充值金额）。
    
-   _(无修饰符)_：**默认写操作**。可读可写，修改状态时需消耗 Gas。
    

## 五、 三大核心安全漏洞与标准防护

### 1\. 重入攻击 (Reentrancy) —— 最致命

-   **漏洞原理**：先向外部地址转账，外部地址通过恶意代码重新调用合约提款函数，此时合约**尚未更新扣款状态**，导致资金被反复提空。
    
-   **防护原则 (CEI 模式)**：**Checks（检查）-> Effects（生效/状态修改）-> Interactions（外部交互/转账）**。
    

```
// ✅ 安全转账范式 (CEI 模式)
function withdraw() external {
    uint256 amount = balances[msg.sender];
    require(amount > 0, "No balance"); // 1. 检查 (Checks)
    
    balances[msg.sender] = 0;          // 2. 生效 (Effects) - 先扣款！
    
    (bool success, ) = msg.sender.call{value: amount}(""); // 3. 交互 (Interactions)
    require(success, "Transfer failed");
}
```

> _注：也可以直接使用 OpenZeppelin 的_ `ReentrancyGuard` _库并加上_ `nonReentrant` _修饰符。_

### 2\. 访问控制漏洞 (Access Control)

-   **漏洞原理**：关键的管理函数（如提款、暂停、升级）没有加修饰符限制，任何人都能调用。
    
-   **防护方式**：使用 `Ownable` 并加上 `onlyOwner` 权限控制。
    

```
address public owner;

modifier onlyOwner() {
    require(msg.sender == owner, "Not owner");
    _; // 执行被修饰的函数体
}
```

### 3\. 整数溢出 (Overflow / Underflow)

-   **漏洞原理**：数值超出边界后回绕（如 `0 - 1` 变为 $2^{256} - 1$）。
    
-   **防护方式**：使用 **Solidity 0.8.0+** 编译器（默认内置了溢出检查，发生溢出时自动回滚交易）。
    

## 六、 链上留言板实战合约模板 (可直接复制)

你可以直接在 [Remix IDE](https://remix.ethereum.org) 中创建 `.sol` 文件并粘贴以下代码进行测试：

```
contract MessageBoard {
    // 1. 状态变量：地址 => 留言数组
    mapping(address => string[]) public messages;
    
    // 2. 事件定义：便于前端/检索器监听
    event NewMessage(address indexed sender, string message);

    // 3. 构造函数
    constructor() {
        messages[msg.sender].push("Hello ETH Pandas!");
    }

    // 4. 写函数 (需要 Gas)
    function leaveMessage(string memory _msg) external {
        require(bytes(_msg).length > 0, "Message cannot be empty");
        messages[msg.sender].push(_msg);
        emit NewMessage(msg.sender, _msg);
    }

    // 5. 读函数 (免费)
    function getMessage(address user, uint256 index) external view returns (string memory) {
        return messages[user][index];
    }

    function getMessageCount(address user) external view returns (uint256) {
        return messages[user].length;
    }
}
```

## 七、 以太坊技术底层

### 1\. 账户模型

-   **EOA 账户（外部拥有账户）**：如小狐狸钱包，由**私钥**控制。有 `balance` 和 `nonce`。**只有 EOA 账户能主动发起交易并付 Gas 费**。
    
-   **Contract 账户（合约账户）**：由**部署的代码**控制。除 `balance` 和 `nonce` 外，还拥有 `codeHash` 与 `storageRoot`。只能由 EOA 间接触发。
    

### 2\. Gas 计费公式 (EIP-1559)

{实际交易费} = {Gas Used} \* ({Base Fee} + {Priority Fee})

-   `Base Fee`（基础费）：由全网根据拥堵情况动态计算，**直接销毁**。
    
-   `Priority Fee`（小费）：支付给验证者以激励快速打包。
    
-   `Gas Limit`：用户设定的本笔交易最大可承载的工作量上限，防死循环。
    

### 3\. 测试网速查

-   **Sepolia**：应用与合约开发**首选测试网**，高度模拟主网。
    
-   **Holesky**：大规模验证者、质押及底层协议测试网。
    

## 八、 高阶内容与工程标准

### 1\. Gas 极致优化三板斧

-   **减少 Storage 写入**：写入 `storage` 成本极高（`SSTORE` approx= 20,000 Gas）。在循环中尽量把数据放到 `memory` 中计算，最后一次性更新到 `storage`（缓存法则）。
    
-   **循环优化**：不要在循环条件中写 `i < arr.length`，应提前缓存：`uint256 len = arr.length; for (uint256 i = 0; i < len; ++i)`。
    
-   **精确打包**：对 struct 变量合理排列，让多个小容量变量（如 `uint128`）合占一个 32 字节槽位（Bit Packing 槽位压缩）。
    

### 2\. 自动化审计三大金刚

-   **Slither**（静态分析）：命令行 `slither .` 快速检测常规漏洞及代码不规范。
    
-   **MythX**（云端检测）：基于云的高级智能合约分析。
    
-   **Foundry Fuzzing**（模糊测试）：使用 `forge test` 的随机输入探测极端边界安全。

### Blockchain

-   去中心化分布式账本（Distributed Ledger）
    
-   Block = 交易数据 + 上一区块 Hash
    
-   Chain = 多个 Block 按时间连接
    

### 特性

-   去中心化（Distributed）
    
-   不可篡改（Immutable）
    
-   公开透明（Transparent）
    
-   匿名（Address ≠ Identity）
    

### 运作流程

User  
→ 发起 Transaction  
→ Broadcast  
→ Validator 验证  
→ 打包 Block  
→ 上链  
→ Validator 获得 Reward + Gas Fee

### 为什么需要 BTC？

-   激励节点维护网络
    
-   支付网络奖励
    

### 三种链

Public Chain

-   人人可加入
    
-   去中心化
    
-   BTC、Ethereum
    

Consortium Chain

-   多机构共同管理
    
-   半开放
    

Private Chain

-   企业内部
    
-   中心化
    

### Web2 vs Web3

Web2

-   Platform owns data
    
-   Trust Platform
    

Web3

-   User owns data/assets
    
-   Wallet + Smart Contract
    
-   Trust Code
    

### Web3 核心

Wallet  
→ Sign  
→ Smart Contract  
→ Blockchain

### 去中心化优点

-   Trustless
    
-   Anti-censorship
    
-   User Ownership
    
-   Open Innovation
    

### 缺点

-   TPS 较低
    
-   Gas 高
    
-   UX 差
    
-   法规不成熟
